This is a great architectural decision. Moving from "clients trying to figure it out" to a Server-Authoritative Spawner System is the professional standard for multiplayer games.

To make this modular for future expansion (like your Gardener idea), we need a Data-Driven approach. Instead of writing code for "Tents", we write code for "Spawners," and we define Tents (or Gardens) as types of spawners in a configuration file.

Here is the modular solution design.

1. The Architecture
We will create a new server-side module: AISpawnerSystem.js.

Config Driven: It reads a list of rules (e.g., "Gardens spawn Gardeners", "Tents spawn Enemies").

Stateful: It remembers which spawner is on cooldown or permanently dead.

Reactive: It watches player proximity to trigger spawns.

2. Step 1: The Configuration (server/SpawnerConfig.js)
This is where the modularity lives. You define the rules here without changing the engine code.

JavaScript
// server/SpawnerConfig.js
const SPAWNER_TYPES = {
    'tent': {
        aiType: 'combat_enemy',   // What class to spawn on client
        maxCount: 1,              // How many AI per structure
        triggerDistance: 20,      // Player must be within this range
        despawnDistance: 50,      // Unload AI if players go this far
        respawnTime: -1,          // -1 = Never respawn (Permadeath), >0 = Milliseconds
        aggro: true               // Metadata passed to client
    },
    'garden': {
        aiType: 'gardener_npc',
        maxCount: 1,
        triggerDistance: 10,
        despawnDistance: 40,
        respawnTime: 60000,       // Respawn after 60 seconds if killed
        aggro: false
    }
    // Future structures go here...
};

module.exports = { SPAWNER_TYPES };
3. Step 2: The Server Logic (server/AISpawnerSystem.js)
This system runs on the server loop. It checks chunks with players in them, looks for structures defined in the config, and tells clients to spawn things.

JavaScript
// server/AISpawnerSystem.js
const { SPAWNER_TYPES } = require('./SpawnerConfig');

class AISpawnerSystem {
    constructor(chunkManager, messageRouter) {
        this.chunkManager = chunkManager;
        this.messageRouter = messageRouter;
        
        // Runtime state: structureId -> { activeAI: [], deadUntil: timestamp }
        // This replaces your client-side "deadTentAIs" set
        this.spawnerStates = new Map(); 
    }

    update() {
        // 1. Get all active chunks (chunks with players)
        const activeChunks = this.chunkManager.getActiveChunks();

        activeChunks.forEach(chunkId => {
            const chunk = this.chunkManager.getChunk(chunkId);
            const players = chunk.players || [];
            
            if (players.length === 0) return;

            // 2. Find structures in this chunk that match our Config
            const structures = this.getStructuresInChunk(chunk);

            structures.forEach(structure => {
                const config = SPAWNER_TYPES[structure.name]; // e.g. 'tent'
                if (!config) return;

                this.processSpawner(structure, config, players, chunkId);
            });
        });
    }

    processSpawner(structure, config, players, chunkId) {
        const state = this.getSpawnerState(structure.id);

        // CHECK: Is it dead/on cooldown?
        if (state.deadUntil === -1) return; // Permanently dead
        if (state.deadUntil > Date.now()) return; // On cooldown

        // CHECK: Is AI already active?
        if (state.activeAI.length >= config.maxCount) return;

        // CHECK: Is a player close enough?
        const isPlayerNear = players.some(p => this.getDistance(p, structure) < config.triggerDistance);

        if (isPlayerNear) {
            // SPAWN IT!
            const aiId = `ai_${structure.id}_${Date.now()}`;
            
            // Mark as active so we don't double spawn
            state.activeAI.push(aiId);

            // Tell ALL clients in range to spawn this specific AI
            this.messageRouter.broadcastTo3x3Grid(chunkId, {
                type: 'spawn_ai_command',
                payload: {
                    id: aiId,
                    type: config.aiType, // 'combat_enemy' or 'gardener_npc'
                    position: structure.position,
                    ownerId: structure.id // Link AI to the tent/garden
                }
            });
            
            console.log(`[Server] Spawned ${config.aiType} at ${structure.id}`);
        }
    }

    // Called when client sends 'ai_death' message
    handleAIDeath(spawnerId) {
        const state = this.getSpawnerState(spawnerId);
        const structureType = this.getStructureTypeById(spawnerId); 
        const config = SPAWNER_TYPES[structureType];

        state.activeAI = []; // Clear active count
        
        if (config.respawnTime === -1) {
            state.deadUntil = -1; // Permadeath
        } else {
            state.deadUntil = Date.now() + config.respawnTime;
        }
    }

    // ... helper methods (getDistance, getSpawnerState, etc) ...
}

module.exports = AISpawnerSystem;

    if (this.activeAIs.has(id)) return;

    let aiEntity;

    }
}
Why this fixes your problems:
Modular: To add a Gardener, you just add 4 lines to SpawnerConfig.js and 3 lines to the client switch statement. No complex logic changes.

Fixes Double Spawning: The Server is the single source of truth. It sends one spawn ID (ai_tent1_12345). If Player A and Player B both receive it, they just instantiate the visual representation of that specific ID.

Fixes Infinite Respawning: The spawnerState map on the server holds the "Dead" status. If Player B joins 10 minutes later, the server checks the map, sees deadUntil: -1, and simply never sends a spawn command for that tent.
